# ハンズオン

## ハンズオン実行環境の準備

### Python 依存ライブラリのインストール

```bash
cd appcodes
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### ユーザーへの同意付与

OBO フロー用のスコープに対してユーザーの同意が必要な場合は、以下のコマンドで同意 URL を生成してブラウザで開きます。

```bash
TENANT_ID="$(azd env get-value AZURE_TENANT_ID)"
CLIENT_ID="$(azd env get-value AZURE_OBO_CLIENT_ID)"
echo "https://login.microsoftonline.com/${TENANT_ID}/oauth2/v2.0/authorize?client_id=${CLIENT_ID}&response_type=code&redirect_uri=http://localhost&scope=api://${CLIENT_ID}/user_impersonation%20https://search.azure.com/user_impersonation"
```

## 各シナリオの概要

### シナリオ 1: ACL なし — Foundry Agent 経由

Foundry Project に事前デプロイされたエージェント (`info-agent-tartaria`) を呼び出します。
ACL は無効のためすべてのドキュメントが参照でき、OBO フロー不要でシンプルに動作を確認できます。

→ [シナリオ 1 を開始する](./シナリオ1_ACLなし.md)

---

### シナリオ 2: ACL あり — Foundry IQ Knowledge Agent

MSAL の OBO フローでユーザー代理トークンを取得し、MCP リクエストの `x-ms-query-source-authorization` ヘッダーに付与します。
Foundry IQ がユーザーのグループに応じてドキュメントをフィルタリングするため、ACL によるアクセス制御を体験できます。

> **⚠️ 既知の問題**: ADLS Gen2 ACL による検索フィルタリング (`x-ms-query-source-authorization`) とハイブリッド検索（セマンティック＋ベクトル）を同時に使用すると、以下のエラーが発生しハンズオンを完遂できない状態です。
> ```json
> {"error": {"code": "", "message": "Unable to complete search request successfully"}}
> ```
> 現時点では回避策を調査中です。シナリオ 2 は参考として確認し、動作確認はシナリオ 3 で行ってください。

→ [シナリオ 2 を開始する](./シナリオ2_ACLあり_FoundryIQ.md)

---

### シナリオ 3: ACL あり — クラシック RAG パターン

Foundry IQ を使わず、Azure OpenAI と Azure AI Search を直接組み合わせます。
JWT をデコードしてグループ ID を抽出し Search の `filter` パラメータに適用することで、シナリオ 2 と同じ ACL 結果を自前実装で実現します。

→ [シナリオ 3 を開始する](./シナリオ3_ACLあり_クラシックRAG.md)
