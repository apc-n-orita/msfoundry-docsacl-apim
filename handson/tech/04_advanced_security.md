# Microsoft Foundry のさらなるセキュリティ強化

## Microsoft Defender for Cloud — AI 脅威保護

### 概要

**Microsoft Defender for Cloud** は Foundry ワークロードに対する脅威検出・セキュリティ体制管理を提供します。  
AI Services プランを有効化することで、Foundry アプリケーションを標的にした攻撃をリアルタイムに検出・アラート通知します。

### 主な機能

| 機能                            | 説明                                                          |
| ------------------------------- | ------------------------------------------------------------- |
| **AI インフラ検出**             | デプロイされたモデル・サービス・AI リソースの統合ビューを提供 |
| **セキュリティ体制管理 (CSPM)** | 設定ミス・脆弱性・攻撃パスを特定し、修復の優先度を提示        |
| **脅威検出**                    | 不審なプロンプト・異常な実行パターンを検出してアラートを生成  |
| **エージェント インベントリ**   | Foundry で作成したすべての AI エージェントを一元管理          |
| **Defender XDR 統合**           | 全 Defender 製品のシグナルを相関分析し、攻撃全体像を把握      |

### Foundry エージェント向けセキュリティ推奨事項

Defender は以下のような高リスクの設定を自動検出します。

| 推奨事項                                                                                      | 重要度 |
| --------------------------------------------------------------------------------------------- | ------ |
| 間接プロンプトインジェクションリスクのあるエージェントは MCP ツールに人間による承認を追加する | High   |
| ガードレールに Jailbreak 制御が設定されていないエージェントを修正する                         | Medium |
| MCP ツールに許可ツールリストを設定する (`require_approval: "never"` の場合)                   | Medium |
| Foundry のデフォルトストレージに Entra ID 認証を使用する                                      | High   |

### 有効化手順

1. Azure ポータルで **Microsoft Defender for Cloud** を開く
2. **環境設定** → 対象サブスクリプションを選択
3. **Defender プラン** で **AI サービス** を **オン** にする

## Microsoft Entra ID — 条件付きアクセス（Agent ID）

### 概要

Microsoft Entra ID の **条件付きアクセス for Agent ID**（プレビュー）は、AI エージェントをゼロトラストの対象として扱い、**エージェント ID ごとにアクセスポリシーを適用**する新しいアイデンティティ制御です。

従来の「送信元 IP 制限」ではなく、**Identity Protection のリスク評価** と **条件付きアクセスポリシー** による動的な制御を行います。

### Agent ID の概念

Entra ID にはエージェント専用のアイデンティティ構造が導入されています。

| 用語                | 説明                                                               |
| ------------------- | ------------------------------------------------------------------ |
| **Agent Blueprint** | エージェントタイプの論理定義（テンプレート）                       |
| **Agent Identity**  | 実際に動作するエージェントインスタンスの ID                        |
| **Agent User**      | ユーザーアカウントが必要なシナリオ（メール・Teams 連携など）で使用 |

### 条件付きアクセスポリシーの適用

ポリシーのスコープをエージェント単位で設定できます。

```
適用対象:
- すべてのエージェント ID
- 特定のエージェント（カスタムセキュリティ属性でフィルタ）
- 個別のエージェント ID
```

**Identity Protection との連携により:**

1. エージェントの異常なアクセス行動（未知リソースへのアクセス、大量サインイン試行）を自動検出
2. 高リスクエージェントを自動ブロック（Microsoft Managed Policies）
3. リスクシグナルを条件付きアクセスに渡し、セッション制御を適用

### サービスプリンシパルとの違い

AI エージェントには従来のサービスプリンシパルではなく **Agent Identity** の使用を推奨します。

| 観点               | サービスプリンシパル         | Agent Identity                     |
| ------------------ | ---------------------------- | ---------------------------------- |
| 監査ログ           | 汎用的なアプリ ID として記録 | エージェント専用エントリとして記録 |
| スポンサーシップ   | なし                         | オーナー（責任者）を必須設定       |
| ライフサイクル管理 | 手動                         | Blueprint 経由で一元管理           |
| 高権限の制限       | なし                         | プラットフォームレベルで制限       |
| 条件付きアクセス   | ワークロード ID ポリシーのみ | エージェント専用ポリシー           |

## Microsoft Purview — データガバナンス強化

### 概要

**Microsoft Purview** を Foundry と統合することで、AI アプリケーションが処理するプロンプトと応答に対して企業のデータガバナンス・コンプライアンスポリシーを適用できます。

Foundry ポータルの **Operate → Compliance → Security posture** から Microsoft Purview トグルを有効化するだけで、対象サブスクリプションの全 Azure AI アプリケーションのデータが Purview に送信されます（Azure AI Account Owner ロールが必要）。

### サポートされる Purview 機能

| 機能                         | 概要                                                                                         |
| ---------------------------- | -------------------------------------------------------------------------------------------- |
| **DSPM for AI**              | AI 利用の検出・セキュリティ態勢管理。ワンクリックポリシーでプロンプト/レスポンスを取得・分析 |
| **監査 (Audit)**             | プロンプト・応答を統合監査ログに記録。いつ・誰が・どのように AI とやり取りしたかを追跡       |
| **データ分類 (SIT)**         | 機密情報タイプでプロンプト/レスポンス内の機密データを検出・分類                              |
| **DLP（データ損失防止）**    | 機密情報を含むプロンプトのブロック（API または Agent Framework で適用）                      |
| **Insider Risk Management**  | プロンプトインジェクション攻撃や保護対象資料へのアクセスなどリスクのある AI 使用を検出       |
| **eDiscovery**               | 法的・規制要件に対応するための AI インタラクションデータの検索・保持                         |
| **データライフサイクル管理** | 保持ポリシー・アーカイブによる AI データの適切な管理                                         |
| **Compliance Manager**       | GDPR / HIPAA などの規制テンプレートに基づくコンプライアンス評価                              |

### ⚠️ Data Security ポリシーの適用範囲

Purview の DLP などの **Data Security ポリシー**が適用されるのは、**`/chat/completions` エンドポイント + Entra ID ユーザーコンテキスト認証**の組み合わせのみです。

| 認証シナリオ                                        | 監査・DSPM     | DLP ポリシー適用 |
| --------------------------------------------------- | -------------- | ---------------- |
| `/chat/completions` + Entra ID ユーザーコンテキスト | ✅             | ✅               |
| それ以外（マネージド ID・API キーなど）             | ✅（閲覧のみ） | ❌               |

> ハンズオンの構成（OBO フロー経由の Foundry Agent API 呼び出し）は「それ以外の認証シナリオ」に該当するため、監査・DSPM での閲覧は可能ですが DLP ポリシーは適用されません。

### 統合方式とシナリオ別の対応

| シナリオ                 | Foundry ネイティブ | Agent Framework | Purview API |
| ------------------------ | ------------------ | --------------- | ----------- |
| 監査・DSPM・データ分類   | ✅（推奨）         | ✅              | ✅          |
| DLP 防止・データ漏洩対策 | ❌                 | ✅              | ✅          |
| データ過剰共有防止       | ❌                 | ❌              | ✅          |

> ※ **Foundry ネイティブでの DLP・データ過剰共有防止の代替策**  
> 現時点で Foundry ネイティブはこれらに非対応ですが、**Azure AI Search × Purview 機密度ラベル連携**を利用することで、ドキュメントに付与された Purview 機密度ラベルをインデックス作成時に自動抽出し、クエリ時にユーザーの READ 使用権と照合してドキュメントレベルのアクセス制御を実現できます。RAG パイプラインでの機密情報フィルタリングに有効です。  
> → [Azure AI Search × Purview 機密度ラベル連携ガイド](../../knowleage/Azure_AI_Search_Purview_Sensitivity_Labels_Guide.md)

詳細は [Microsoft Foundry × Microsoft Purview 連携ガイド](../../knowleage/Microsoft_Foundry_Purview_Integration_Guide.md) を参照してください。
